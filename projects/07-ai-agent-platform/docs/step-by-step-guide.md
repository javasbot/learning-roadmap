# 前端 AI Agent 交互编排架构

> 如果说 RAG 是对已有知识库的查询，那 Agent 就是给大模型**加上了“眼睛、手和键盘”**。大模型可以通过输出特定的格式符（比如 JSON）命令你（指代浏览器或物理机前端系统）去代替它发起一次天气 API 请求，或者读写一下本地的文件。

## Agent 状态机的重要性（Why State Machine?）
在大模型未崛起之前，网页大部分业务的生命周期极其简单：`Loading` -> `Success(Render)` 或者 `Failed`。这叫典型的两段式。

**但在由大模型主导的协同过程中**，你告诉大模型：“分析一下我电脑桌面上的 PDF 财报并得出今年是否应该加大投资的利润论点”。
此时大模型脑子里的流转节点极多：
1. `【思考】` - 它意识到自己是个模型看不见桌面。
2. `【调用工具】` - 它命令你前端，去开启 OS 文件权限拿取 PDF。
3. `【解析意图/工具】` - 它命令你调用另一个插件把 PDF 拆成 1 万个字再用 Chunk 返给它。
4. `【再思考】` - 它看了这两万字，可能还需要你帮他画个饼图。
5. `【工具再调用】` - 你前端接到了指令并用 Echarts 渲染了一个饼状图数据存在本地。
6. `【流式作答】` - 最终结合图表数据吐出 Markdown 文章。

如果你前端用最常规的 `loading=true` `await res` 去走这条线，光中继报错、请求超时重拾或者是多工具并发的竞态问题，就会把整个交互撕扯得粉碎。
这就是为什么所有的一线 AI 企业，都在用**状态机 (FSM)** 管理大模型的编排。在我们的项目中你可以清晰的看到从 `IDLE` 到 `GENERATING_RESPONSE` 的流转管控架构体系。

## 前端的交互革新：卡片可视化
像 AutoGPT, 之前的 OpenAI 的界面一样，当 Agent 在帮你干很繁杂的背锅活（如网络请求、看 PDF）时，用户由于看不见而会极度焦躁。

优秀的 P10 体验设计师/前端架构，这时候绝对不是粗暴的一个转圈圈。而是会根据上述状态机流转到 `CALLING_TOOL` 的瞬间，在瀑布流里高仿生成一张精致的卡片（如 `ui/tool-visualizer.js` 里做的一样）。这张卡片甚至能透传打印出正在运行的 Shell 指令。既提供了心理慰藉安抚了用户的漫长等待，又在 UI 上展示了极强的专业感（这被称为过程白盒透明化机制）。这也是大厂如 Coze 都在重金投入的方向！
