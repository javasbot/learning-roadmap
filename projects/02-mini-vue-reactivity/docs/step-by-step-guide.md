# Mini Vue Reactivity 系统源码解析

> 面试高频问题：Vue3 的响应式原理是怎样的？为什么放弃 `Object.defineProperty` 而使用 `Proxy`？计算属性的“懒计算”和脏值检测是如何做到的？什么是响应式丢失问题（解构为什么会失去响应）？

## 背景剖析
Vue 最引以为豪的核心竞争力就是其**几乎零配置**的数据响应（Reactivity）系统。而从 Vue 2 到 Vue 3 的跃变，正是基于 ES6 新标准的一场大换血。

### Vue 2 (`Object.defineProperty`) 的死穴：
1. **监听粒度只能在已经定义的固有“key”身上**：不能检测对象属性的添加或删除（所以才有可恶的 `$set` / `$delete`）。
2. **对数组的绝望**：它甚至不能检测基于下标索引的修改 `arr[0] = 1`。尤雨溪只能暴力拦截重写原生的 `push`、`slice`、`pop` 等七个数组方法。
3. **初始化性能噩梦**：不管这个 5000 行嵌套 JSON 里的数据首屏用没用到，Vue2 只要拿到了，就要在 `created` 前用深度递归把所有的节点全部 `defineProperty` 获取 `set/get`，导致首屏卡死。

## 攻坚环节一：`reactive.js` 的诞生
利用 `Proxy`，你是在代理“**引用/对象（Target）**本身”，而不是像以前一样盯着每个属性的小尾巴。
```javascript
const p = new Proxy(target, { get, set, deleteProperty })
```
只要有人想操作在这个对象的**任何**部分（哪怕是突然塞进去一个过去并不存在的 `age`），`Proxy` 全都能管得到。
而且他是**懒绑定**的。如果里面有一千层，你不去 `res.age.girlfriend` 访问到深处，他压根不会对底部的嵌套对象调用递归封包去转换 `reactive`。大幅缩短首屏白屏时间。这就是 Vue 3 快如闪电的原因。

## 攻坚环节二：`effect.js` - WeakMap 与 闭包引擎
响应式的秘诀就是两个字：**收集** 和 **触发**。
当你 `{{ count }}` 访问的一瞬间，我们要把“更新视图”的那个回调函数给揪出来，存进数据结构里去！这就是 `track`（追踪）。

这种结构是：`WeakMap` -> `Map` -> `Set(effectFN)` 
我们把整个系统的数据和回调，紧密关联在这个三维网络中。如果根对象被垃圾回收销毁，`WeakMap` 也会自动清除避免内存泄漏。

然后在你修改数据的一刹那 `count++`。底层的 Proxy 拦截器捕获到了写动作，火速把上述小本本里面因为 `count` 这个特定 key 而存在的 `Set` 函数堆，全数拿出来跑一遍叫醒它们（`trigger`）。它们跑的时候就会重绘修改真实的 DOM，以此完成了神奇的数据驱动视图的转换。

## 最终环节：`computed.js` 到底是什么？
面试喜欢问各种响应式的特性。
`computed` 看似高深莫测，其实它不过是有两个非常傲娇标签的特殊 `effect()`：
1. `lazy: true`: 别一上来就运行里面的算术题。
2. 有自己的 `scheduler`: 一旦上游数值改了，别急着把所有的值全部重算一遍。我的调度器只要把内置表示状态缓存是否陈旧失效的那个开关（也就是源码里的 `dirty` 变量）狠狠地变回 `true`（脏数据）即可。

等什么时候真实组件在页面渲染时试图去读取它，它的读取器捕捉到这个绿光 `dirty=true`，这时候他才说：“噢，我确实需要算算数了，重新在我的 `eff()` 里执行一回。顺带再挂上不脏的红牌子”。

这个在架构上叫做按需（Lazy evaluation）缓存设计。

---

> [!TIP]
> **深入反思：解构丢失陷阱**  
> 这也是 P7 面试极其喜欢考的地方：“为什么我们使用 `const { count } = state` 把它解构出来以后，页面就不再有响应了？”
> 因为你这个时候赋值出去给组件模板使用的 `count` 已经不再是一个对象的强引用！它是一个光秃秃的字符串或整形值栈内存拷贝！
> 没了包裹一切监控动作的外壳代理 `Proxy`，你直接改这个解构的数值，Vue 上哪知道去？这也是为什么需要 `toRefs` 从设计层面上硬把这个光脱脱的值用 `{ value }` 给包回到 `Ref` 对象身上的原因。
