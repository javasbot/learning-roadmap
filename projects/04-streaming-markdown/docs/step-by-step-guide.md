# 流式内容引擎 (Streaming Markdown)

> 为什么要做这个？在 ChatGPT 出来以前，传统的网页都是服务器“算完打包”直接返回全量 JSON。在 LLM 时代，模型是按 Token （也就是只言片语）通过 Server-Sent Events (SSE) 长链接一点点吐出来的。

## 难题一：残缺 DOM 树（解析容错）
假设大模型刚说到：`这是一段代码 \`\`\`javas` 本网卡住了。
这在原生 `marked.js` 等工具里是极其致命的。因为如果你直接套正则去处理，由于它没有找到结尾的 ` \`\`\` `，它会把接下来一直到最后几十首诗歌全当成代码丢进一块灰色的框里，导致排版大范围雪崩。

**解决策略：**
我们必须在我们的 `StreamingParser` 里维护一个**状态机（State Machine）**。
记录一个 `isInCodeBlock` 的全局变量。当只遇到一个开口符号而当前字符枯竭时，框架必须**无条件、强制主动**给他加上 `</code></pre>` 的闭合标签，并且在其中插入一个 `█` 的闪烁光标假借位。等大批的下一波 Token 到来，再把这段补上的强制退回去刷新重新渲染。

## 难题二：渲染狂飙（卡帧掉帧）
大模型速度快的时候，10 毫秒给你丢一个字母回来。前端接收到 50 个字母，如果触发了 50 次类似于 React `setState` 或者是 `innerHTML = result` 的操作。浏览器主线程需要强制计算 CSS 和树结构，用户如果恰巧想滚动鼠标滚轮去看历史消息，整个界面会彻底冻结无法使用。

**解法：时间切片与打批 `requestAnimationFrame` (rAF)**
无论你往 `renderer.receiveToken` 里面用多快的速度（哪怕是一瞬间抛 100 遍请求）抛数据，它都不会立刻执行更新。
它只会：
1. 更新本地的字符串 buffer。
2. 将真实的写 DOM 动作丢给 `requestAnimationFrame`。

由于 rAF 是浏览器内置的、在每次显示器即将刷新下一帧图像（也就是 16.6ms 左右空闲期）才会去执行的 API。所以上述 100 次的狂轰滥炸，在这里会被“合并打批”成 1 次极其干净利落的 DOM 修改。这是 P8 大厂前端处理高频实时流数据必须展现出的基本功！
