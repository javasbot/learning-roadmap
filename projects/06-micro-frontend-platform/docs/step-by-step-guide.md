# 微前端基座容器系统解析

> 面试高频问题：Qiankun 的核心运行机制是什么？为何使用 HTML Entry？JS JS Sandbox 的内部怎么实现的？CSS 该如何隔离？

## 背景剖析
当我们维护一个巨石应用（庞大的 B 端 ERP、中台），如果业务横跨 6、7 个部门甚至不同的技术栈，所有代码揉杂在一起打包往往慢达到 20 分钟以上，一个人提交错误就会引发雪崩级发布阻塞。所谓微前端，**就是像后端微服务一样，允许各自业务团队使用各自语言各自独立发版，但最终呈现在用户眼里是“一个完美的、连贯的网页”**。

## 方案 1：iframe 时代的辛酸
如果只是追求独立组合，使用几个 `iframe` 拼在一块不就行了？绝不行。
1. **弹窗噩梦**：iframe 里的弹窗由于边界的存在，无论怎么弄永远只能在页面的一个小小框里弹出来，没法全屏遮罩居中。
2. **路由地狱**：点来点去返回不了，URL 状态和 `history.pushState` 被彻底撕裂。
3. **极差性能**：每一次切换几乎等同于重新打开一次新的 Chrome Tab 页。

这也是为何我们必须打造类似 **Single-Spa** 与 **Qiankun** 的解法。

## 方案 2：我们手写的 HTML Entry 模型（Loader 解读）
在早期（如 single-spa）我们需要每个子应用自己把自己导出一个名为 `bootstrap`, `mount`, `unmount` 的大对象然后注入进 JS 文件里（这叫 JS Entry）。但是对已经运行了多年的老应用非常的不友好。

现代框架基本使用类似该工程下 `container/loader.js` 的实现：
主应用直接 Fetch 下载远端子应用的 `index.html`。（比如 localhost:8080 发起 fetch 去拿 localhost:8081）。拿到长长的一串文本后，主框架通过正则将其中的 `<link>` (CSS)、`<script>`(脚本资源)、注释、乃至正文的 HTML DOM **肢解和提取**。
1. 把提取出来的 DOM 插到自己的容器 `<div id="sub-app">` 内。
2. 利用 `new Function` 把提取出来的 JS 以闭包形式加载执行。

## 方案 3：防污染护盾 (JS Sandbox 解析)
如果我们这么直接把别人的代码拽进来我们的页面执行，万一人家在他的应用逻辑深处写了一行 `window.globalVariable = '污染你'` 怎么办？你会引发无尽的全局变量互相覆盖。

通过阅读 `container/sandbox.js` 你可以看到业界最优解法：**Proxy Sandbox**。
在解析完脚本要执行的那个闭包里：
```javascript
new Function('window', `with(window) { ${远程拉来的代码} }`)
```
我们在外面构建了一个 `fakeWindow`，并借由 `Proxy` 进行 `set` 拦截。
当代码内部尝试修改他自认为的全局时。`Proxy` 温柔地接过了数据，把它存放在了自己的**假 Window**里兜住，并在之后提供原路存取服务。整个过程中主真实 Window 没有收到哪怕一丝一毫的影响！

## 其他附加面试考点：CSS 如何隔离？
此项目侧重于 JS 沙箱的实现。但是面试如果你需要解答，请记住两个主要道路：
1. **工程化类名（强约定）**：子应用开发时，所有人都必须用 CSS Modules 或者 BEM 给自己加命名空间。
2. **Shadow DOM**：原生的终极强隔离属性！创建了一个封闭的渲染黑洞空间（比如现在的 `<video>` 原生浏览器录像器）。但在应对如 Ant Design 动态把弹框 Append 到 body 层时，容易出不显示或者穿透遮罩的问题。
